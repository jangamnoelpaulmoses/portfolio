The guy who built Claude Code doesn't prompt it anymore and if you're on job-search this is for you!
Boris Cherny, head of Claude Code at Anthropic: "I don't prompt Claude anymore. My job is to write loops."
It's called loop engineering, and the shift is simple: stop chatting with AI, start deploying it as a worker that runs while you walk away.

A Zapier automation is a light switch, one trigger, one action, done. A loop is a guard dog. It watches, remembers what it already saw, and only barks when it matters.

You build one by answering 5 questions:
 What should it watch?
 How often should it check?
 What exact change is it hunting for?
 What does it do the moment that happens?
 When should it stop and tap a human?

For example, if you're on job-search, this can be insanely useful...a job-hunter loop on Claude's /loop!

Feed it your target companies and filters. It spawns sub-agents to scan the boards in parallel, 3x a day. Remembers what it already showed you (no duplicates), drafts a tailored cover letter for each real match, and hands you a dashboard to approve. Runs on Anthropic's cloud, so your laptop can be off.
But here's the rule that separates a useful loop from a disaster, the Human Gate:

The more powerful the loop, the stricter its boundaries. Anything touching your money, reputation, or customers freezes and waits for a human click. The loop watches and drafts. It does NOT buy the shoes, send the cold email, or submit the application on its own.

Prompt engineering taught us how to talk to AI.
Loop engineering teaches us how to hand off the work.

If you want more hands-on breakdowns on AI agents and system design, follow me, I'll keep shipping them.

BrainOS basically:
pulls knowledge from all those fragmented sources

Imagine you are the lead maintainer of a GitHub repo that millions of developers trust your package and it's hijacked.

No code changes. No breaking updates.
Just one small dependency added.

Just today, there has been a yet another supply chain attack on Axios (used in 100M+ projects weekly), and it can be in your average npm project.

An attacker gained access to a maintainer’s npm account using a long-lived token, giving full control over package releases.

Instead of modifying core code, they added a single dependency (plain-crypto-js) to package.json as shown in the first image, designed to run a postinstall script.

On installation, the script:
• detected the OS (Mac/Windows/Linux)
• downloaded a Remote Access Trojan (RAT) in ~1 second
• deleted itself to remove traces

No errors. No visible signals. Affected versions: 1.14.1 and 0.30.4

How to Check Your System?
- Verify Dependencies:
npm list axios
npm list -g axios and visit https://lnkd.in/g48_Srng for more.
- Run deeper scans across your system to check for any remnants

If affected, treat the machine as fully compromised. Rotate all credentials immediately and prevent your system from automatically installing any version of a package that has been published to the npm registry within the last 3 days
- npm config set min-release-age 3

If you’re interested in real-world engineering insights like this ranging from AI tools, Fullstack, system design to security pitfalls, follow me.

You’ve probably used this package before and it's now compromised.

This week was a reminder that in AI, your system doesn’t break at the model layer. It breaks at the layer you don’t question.

LiteLLM, a library many of us use in production, was compromised via a supply chain attack.

Not a vulnerability. Not a bad config.
A malicious package shipped directly through PyPI.

Here’s what actually happened:
A .pth file was injected into the package. And in Python, .pth files execute automatically on startup. No import needed. No function call.
Just… installed = executed.

Inside that file?
A base64-encoded payload that:
 - Collected environment variables (API keys, tokens)
 - Read SSH keys, cloud creds, kube configs
 - Grabbed CI/CD secrets, docker configs, even shell history
 - Encrypted everything
 - Exfiltrated it to an attacker-controlled endpoint 

And the craziest part?
This didn’t need prompt injection or jailbreaks.
Just a dependency update.
Within 45 minutes, the compromised versions (litellm 1.82.7 and 1.82.8) were downloaded ~47,000 times.

This feels very similar to what I saw while experimenting with agent systems.
To make AI useful, we give it: more tools, access and permissions

But every layer we add becomes another attack surface.

The takeaway isn’t
 “don’t use LiteLLM”
Dependencies are no longer just utilities, they’re part of your threat model.

For those affected, this isn’t a simple uninstall. You have to assume breach, rotate all credentials, revoke access, audit logs, and rebuild affected environments.

Going forward, AI infrastructure needs to be treated like critical infrastructure, where version pinning, isolation, and least-privilege access are baseline, not optional.

We spent the last 2 years asking:
 “How powerful is the model?”
Now the real question is:
 “How secure is everything around it?”
..and definitely a lot more about what runs between the code and the model.
Sources: BigQuery PyPI dataset, futuresearch.ai

This is the production horror story nobody mentions about real-time AI. Yes, the queue always wins.

Here’s the scaling problem I keep seeing:

Let's say you build an AI Insights feature,
“Summarize the last 1 hour”
“Alert me when anomalies happen”
At low traffic, the naive flow looks fine:
API -> DB query -> build prompt -> LLM call -> response

But at scale, especially with multi-tenant or 10k+ RPS, it turns into a queueing nightmare. DB gets hammered by fan-out reads. LLM latency causes request pileups. Retries multiply load. Costs explode because the same “summary” gets recomputed 100 times.

What fixed it for me wasn’t a better model. It was simply better system design.
To actually survive traffic incorporate the below flow

Ingestion lane: push events into Kafka or any PubSub model, let consumers normalize, and store raw data in S3/GCS. The goal is to keep the firehose out of your req,res path so your API isn’t doing heavy lifting under load.

Precompute lane: don’t generate “insights” from scratch on every request. Precompute rolling aggregates per tenant and time window (5 min / 1 hr / 1 day): top issues, deltas, spikes, and representative samples. This keeps the prompt structured, bounded, and cheap.

Serving lane: when a user requests insights, serve cached features + cached insights first, and only call the LLM when it’s truly needed. Cache with a key like tenant + timeframe + insightType, and match TTL to the window so it stays fresh without thrashing.

This is the difference between:
“cool demo”
vs
“production system that doesn’t melt at 9am Monday haha”

Exploded to 100,000+ GitHub stars in under 3 months.

Over the last few days, I’ve been experimenting hands-on with OpenClaw.ai that has lately been the hype and breaking records on GitHub, not as a demo, but as a real, running agent on my own VPS.

Yes, it’s an agent that actually executes and lives on your hardware, connects via WebSockets. Furthermore, it orchestrates real actions using LLMs (GPT-4 / Claude), browser automation, filesystem access, cron jobs, and external APIs.

But using it in practice also makes the tradeoff clear: To be useful, agentic AI needs broad permissions and that creates a massive attack surface. 

This isn’t just a Moltbot issue, it’s a core challenge of agentic AI.
We’re moving from chatbots to systems that act, and the security model hasn’t fully caught up yet.

If you'd like to try it yourself, I'd highly recommend you experimenting on a VPS or an isolated machine and avoid running this on your pc.

My takeaway:
For technically experienced builders who understand VPS isolation, credential hygiene, and risk this is a powerful glimpse into the future.
The future of AI assistants is clearly agent based but the hard part now is making them safe enough to trust by default.
Still experimenting. Still learning. And watching this space very closely.

You’ve probably used this package before and it's now compromised.

This week was a reminder that in AI, your system doesn’t break at the model layer. It breaks at the layer you don’t question.

LiteLLM, a library many of us use in production, was compromised via a supply chain attack.

Not a vulnerability. Not a bad config.
A malicious package shipped directly through PyPI.

Here’s what actually happened:
A .pth file was injected into the package. And in Python, .pth files execute automatically on startup. No import needed. No function call.
Just… installed = executed.

Inside that file?
A base64-encoded payload that:
 - Collected environment variables (API keys, tokens)
 - Read SSH keys, cloud creds, kube configs
 - Grabbed CI/CD secrets, docker configs, even shell history
 - Encrypted everything
 - Exfiltrated it to an attacker-controlled endpoint 

And the craziest part?
This didn’t need prompt injection or jailbreaks.
Just a dependency update.
Within 45 minutes, the compromised versions (litellm 1.82.7 and 1.82.8) were downloaded ~47,000 times.

This feels very similar to what I saw while experimenting with agent systems.
To make AI useful, we give it: more tools, access and permissions

But every layer we add becomes another attack surface.

The takeaway isn’t
 “don’t use LiteLLM”
Dependencies are no longer just utilities, they’re part of your threat model.

For those affected, this isn’t a simple uninstall. You have to assume breach, rotate all credentials, revoke access, audit logs, and rebuild affected environments.

Going forward, AI infrastructure needs to be treated like critical infrastructure, where version pinning, isolation, and least-privilege access are baseline, not optional.

We spent the last 2 years asking:
 “How powerful is the model?”
Now the real question is:
 “How secure is everything around it?”
..and definitely a lot more about what runs between the code and the model.
Sources: BigQuery PyPI dataset, futuresearch.ai

This is the production horror story nobody mentions about real-time AI. Yes, the queue always wins.

Here’s the scaling problem I keep seeing:

Let's say you build an AI Insights feature,
“Summarize the last 1 hour”
“Alert me when anomalies happen”
At low traffic, the naive flow looks fine:
API -> DB query -> build prompt -> LLM call -> response

But at scale, especially with multi-tenant or 10k+ RPS, it turns into a queueing nightmare. DB gets hammered by fan-out reads. LLM latency causes request pileups. Retries multiply load. Costs explode because the same “summary” gets recomputed 100 times.

What fixed it for me wasn’t a better model. It was simply better system design.
To actually survive traffic incorporate the below flow

Ingestion lane: push events into Kafka or any PubSub model, let consumers normalize, and store raw data in S3/GCS. The goal is to keep the firehose out of your req,res path so your API isn’t doing heavy lifting under load.

Precompute lane: don’t generate “insights” from scratch on every request. Precompute rolling aggregates per tenant and time window (5 min / 1 hr / 1 day): top issues, deltas, spikes, and representative samples. This keeps the prompt structured, bounded, and cheap.

Serving lane: when a user requests insights, serve cached features + cached insights first, and only call the LLM when it’s truly needed. Cache with a key like tenant + timeframe + insightType, and match TTL to the window so it stays fresh without thrashing.

This is the difference between:
“cool demo”
vs
“production system that doesn’t melt at 9am Monday haha”

Built WanderAI at the HUD Frontier / RSI RL Environments Hackathon hosted by Y Combinator and HUD this weekend with Aditya Gouroju and Noel Paul Moses J

The idea: a scene-agnostic, object goal navigation policy. Drop it into any room it's never seen like kitchen, living room, anything and it has to find a red ball using only egocentric first-person perception, then generalize that search behavior to brand new scenes.

Traditional llms or SFT models struggles with exploration in unseen environments. Our bet is that episodic reinforcement fine-tuning (RFT) can encode path-tracking behavior directly into model weights, making generalization tractable.

How it works:
• A scene summarizer feeds the agent a symbolic egocentric view as text each step
• Three actions: move forward, turn left, turn right (30° angle)
• Reward is geodesic distance, optimal path around obstacles, not straight-line so the agent learns to actually navigate, not just beeline
• Fine-tuned on Fireworks via episodic RFT; the trained model visibly outperforms the base LLM on path tracking in our 2D demo

Future scope: vision models for action-taking, 3D planar surface detection, and a mapping module with memory constraints.
